# Explain Skill: Card Structure Redesign

## Summary

Реорганизация explain skill вокруг единого шаблона -- "Карточка объяснения" с 6 полями. Текущие 5 Core Principles не удаляются, а вливаются в определения полей карточки. Двухэтапная формула (факты -> понимание) остаётся фундаментом.

## Motivation

Текущий explain skill имеет 5 отдельных принципов, которые нужно помнить и комбинировать. На практике хорошо заходит структура из CLAUDE.md для объяснения инструментов:

- Что / Простыми словами / Зачем / Связи / Альтернативы / Когда пригодится

Эта структура -- по сути переформулировка тех же принципов, но в виде единого шаблона. Карточка работает не только для терминов, но и для решений, подходов, проблем.

## Design

### Архитектура: двухэтапная формула + карточка

```
Этап 1: ЧТО ПРОИСХОДИТ (факты, термины, жаргон как есть)
         ↓
Этап 2: ЧТО ЭТО ЗНАЧИТ (каждая концепция разбирается по карточке)
```

Золотое правило "сначала факт, потом понимание" остаётся над всей структурой -- это порядок работы, не поле карточки.

### Карточка объяснения: 6 полей

| Поле | Что в нём | Впитывает из старого |
|---|---|---|
| **Что** | название, категория, что делает (техническим языком) | Принцип #1 (факт, потом объяснение) |
| **Простыми словами** | одна яркая бытовая аналогия + этимология если помогает | Принцип #2 (термин + аналогия + этимология) |
| **Зачем** | какую проблему решает, причина -> следствие | Принцип #3 (причины и следствия) |
| **Связи** | от чего зависит, что зависит от него, этаж абстракции, связанные термины | Принцип #5 (слои абстракции) |
| **Альтернативы** | что ещё есть, трейд-оффы выбора (получили/потеряли) | Принцип #4 (трейд-оффы) |
| **Когда пригодится** | конкретный сценарий из практики (мини-история) | Новое |

### Инструкции по заполнению полей

**Что** -- название, категория, что делает
- Техническим языком, не упрощая. Термин на английском + перевод/транслитерация
- Указать категорию (runtime, package manager, design pattern, architectural approach...)
- Это "исходный код" объяснения -- реальность как она есть

**Простыми словами** -- аналогия из повседневной жизни
- Одна яркая аналогия, не три размытых
- Этимология если помогает ("cache от франц. cacher -- прятать")
- Аналогия должна быть точной: если она ломается при углублении -- заменить

**Зачем** -- какую проблему решает
- Причина: что было плохо без этого
- Следствие: что стало лучше с этим
- Формат "было -> стало" работает лучше всего

**Связи** -- место в экосистеме
- От чего зависит (prerequisites)
- Что зависит от него (dependents)
- На каком "этаже" абстракции живёт (физика -> код -> система -> пользователь)
- Связанные термины (где ещё встретишь это слово)

**Альтернативы** -- что ещё существует и почему выбрали это
- 1-2 альтернативы, не каталог
- Трейд-оффы выбора: что получили / что потеряли
- Не "X лучше Y", а "X лучше для Z, Y лучше для W"

**Когда пригодится** -- конкретный сценарий
- Реальная ситуация, не абстракция
- "Ты делаешь X, случается Y, и тогда это спасает" -- формат мини-истории

### Правило масштабирования

Карточка гибкая -- масштабируется по сложности объясняемого:

- **Крупная концепция** (инструмент, архитектурный паттерн, технология) -- все 6 полей
- **Средняя** (решение, подход, проблема) -- 3-4 ключевых поля по ситуации
- **Мелкая** (термин в проходящем объяснении) -- "Что" + "Простыми словами", остальное если действительно полезно

### Применимость

Карточка работает для любых концепций, не только терминов:
- Инструменты и технологии
- Архитектурные решения
- Подходы и паттерны
- Проблемы и баги
- Технические решения

## Changes to SKILL.md

### Удаляется

- **Core Principles** (секция с 5 принципами) -- содержание влито в определения полей карточки
- **Vocabulary Builder** (таблица терминов и аналогий) -- статичная, LLM знает эти аналогии

### Добавляется

- **Карточка объяснения** (новая секция, заменяет Core Principles):
  - Шаблон 6 полей
  - Инструкции по каждому полю
  - Правило масштабирования

### Обновляется

- **Explain Formula** -- описание Этапа 2 ссылается на карточку вместо списка "аналогия, причина, следствие..."
- **Пример двухэтапного объяснения** -- переписывается с карточками внутри Этапа 2
- **Examples of Good Explanations** (Docker, Git) -- переписываются в формате карточек
- **Checklist** -- обновляется под поля карточки

### Без изменений

- **Overview** (цель, золотое правило, зачем термины)
- **Conversation Summary Format**
- **Anti-Patterns**

## Итоговая структура SKILL.md

```
1. Frontmatter (name, description)
2. Overview (цель, золотое правило, зачем термины)
3. Карточка объяснения (НОВОЕ)
   - Шаблон 6 полей
   - Инструкции по каждому полю
   - Правило масштабирования
4. Explain Formula (двухэтапная структура)
   - Этап 1 без изменений
   - Этап 2 ссылается на карточку
   - Обновлённый пример
5. Conversation Summary Format
6. Examples of Good Explanations (переписаны с карточками)
7. Anti-Patterns
8. Checklist (обновлён под карточку)
```

## Example: Updated Stage 2 with Cards

```
=== ЭТАП 1: ЧТО ПРОИСХОДИТ ===

Мы подняли Redis как кэш-слой перед PostgreSQL.
API сначала проверяет cache hit в Redis (TTL 5 минут),
при cache miss идёт в PostgreSQL и записывает результат в Redis.
Это уменьшает latency с ~50ms до ~2ms для горячих данных
и снижает нагрузку на primary database.

=== ЭТАП 2: ЧТО ЭТО ЗНАЧИТ ===

**Redis**
- Что: in-memory database (база данных в оперативной памяти)
- Простыми словами: блокнот на столе vs папка в шкафу --
  доступ мгновенный, но места мало
- Зачем: обычная БД хранит данные на диске, это надёжно но медленно.
  Redis держит всё в RAM -- в сотни раз быстрее
- Связи: работает перед PostgreSQL (основной БД), используется
  приложением через клиентскую библиотеку (ioredis, redis-py)
- Альтернативы: Memcached (проще, только key-value),
  встроенный кэш приложения (не расшарить между серверами)
- Когда пригодится: у тебя страница грузится 3 секунды,
  профилируешь -- 80% времени уходит на одни и те же запросы к БД.
  Ставишь Redis, эти запросы отдаются за миллисекунды

**TTL (Time To Live)**
- Что: срок жизни записи в кэше
- Простыми словами: срок годности на упаковке -- после него
  "продукт" выбрасывается и готовится заново
- Зачем: без TTL кэш показывал бы устаревшие данные вечно.
  Короткий TTL = свежие данные но больше промахов,
  длинный = быстро но данные могут устареть

**Cache hit / cache miss**
- Что: нашли ответ в кэше (hit) или нет (miss)
- Простыми словами: открыл шпаргалку -- ответ есть (hit),
  ответа нет -- придётся считать самому (miss)
```

Масштабирование в действии:
- Redis -- полная карточка (6 полей), крупная концепция
- TTL -- 3 поля (Что / Простыми словами / Зачем), средний термин
- Cache hit/miss -- 2 поля (Что / Простыми словами), мелкий термин
