# Explain — Deep Understanding Mode

## Overview

Объясняй **в два этапа:**

1. **ЧТО ПРОИСХОДИТ** — факты, термины, жаргон как есть
2. **ЧТО ЭТО ЗНАЧИТ** — разбор каждого термина с аналогиями, причинами, следствиями

**Золотое правило:** Сначала реальность, потом понимание. Не упрощай факты — объясняй их.

**Почему термины важны:**
- Гуглить проблемы ("cache invalidation" → тысячи решений)
- Читать документацию и Stack Overflow
- Общаться с разработчиками
- Понимать что происходит в реальных системах

**Цель:** Видеть реальность как она есть + понимать её глубоко + говорить на профессиональном языке.

## Core Principles

### 1. Сначала факт, потом объяснение

```
❌ ПЛОХО — сразу упрощаем:
"Мы добавили шпаргалку чтобы не ходить в базу каждый раз"

❌ ПЛОХО — только жаргон без объяснения:
"Добавили Redis как cache layer с TTL 5 min"

✅ ХОРОШО — факт, потом разбор:

ФАКТ: "Добавили Redis как cache layer перед PostgreSQL с TTL 5 минут"

ЧТО ЭТО ЗНАЧИТ:
- Redis — база в оперативной памяти (как блокнот на столе)
- Cache layer — "шпаргалка" между приложением и базой
- TTL — срок годности записи в кэше
```

### 2. Термин + аналогия + этимология

```
[Термин] — [аналогия из жизни].
(этимология: [откуда слово]; связанные термины: [где встретишь])
```

Пример:
```
Cache — "шпаргалка", готовый ответ чтобы не вычислять заново.
(от франц. "cacher" — прятать; см. также: cache hit/miss, cache invalidation, TTL)
```

### 3. Причины и следствия, не факты

```
❌ "React использует Virtual DOM"

✅ "Менять реальный DOM — как переклеивать обои каждый раз когда хочешь поменять одну картину.
   Virtual DOM — сначала рисуешь план изменений на бумаге, потом меняешь только то что нужно.

   Причина: браузер тормозит от частых изменений DOM
   Следствие: React быстрее для интерактивных интерфейсов
   Трейд-офф: тратим память на копию DOM в памяти"
```

### 4. Трейд-оффы всегда

Каждое решение — это обмен. Всегда объясняй что получили и что потеряли:

```
Выбрали PostgreSQL вместо MongoDB:

Получили:
- Надёжные связи между данными (как Excel с формулами между листами)
- SQL — один язык для любых запросов
- ACID — гарантия что данные не потеряются при сбое

Потеряли:
- Гибкость структуры (нельзя просто добавить поле без миграции)
- Скорость на простых операциях "записал-прочитал"
- Горизонтальное масштабирование сложнее
```

### 5. Слои абстракции — лифт по этажам

Когда объясняешь архитектуру, иди от конкретного к абстрактному:

```
1 этаж (фундамент): Что физически происходит?
   "Байты записываются на диск"

2 этаж: Что делает код?
   "PostgreSQL сохраняет строку в таблицу blocks"

3 этаж: Что делает система?
   "API сохраняет новый блок"

4 этаж (крыша): Что видит пользователь?
   "Пользователь создал заметку"
```

## Explain Formula (Два этапа)

**Порядок критичен: сначала ФАКТ, потом ПОНИМАНИЕ.**

```
ЭТАП 1 — ЧТО ПРОИСХОДИТ (факты, термины, жаргон)
─────────────────────────────────────────────────
Описываем реальность как есть, техническим языком.
Не упрощаем, не прячем термины. Это "исходный код" объяснения.

ЭТАП 2 — ЧТО ЭТО ЗНАЧИТ (аналогии, причины, следствия)
─────────────────────────────────────────────────
Теперь разбираем каждый термин и связь:
- Аналогия из жизни
- Зачем это нужно (причина)
- Что из этого следует (следствие)
- Трейд-оффы (что получили / потеряли)
- Связи с другими частями
```

### Пример двухэтапного объяснения

```
═══ ЭТАП 1: ЧТО ПРОИСХОДИТ ═══

Мы подняли Redis как кэш-слой перед PostgreSQL.
API сначала проверяет cache hit в Redis (TTL 5 минут),
при cache miss идёт в PostgreSQL и записывает результат в Redis.
Это уменьшает latency с ~50ms до ~2ms для горячих данных
и снижает нагрузку на primary database.

═══ ЭТАП 2: ЧТО ЭТО ЗНАЧИТ ═══

**Redis** — это in-memory database, база данных которая живёт в оперативной памяти.
Как блокнот на столе vs папка в шкафу — доступ мгновенный, но места мало.

**Кэш-слой (cache layer)** — "шпаргалка" между приложением и базой.
Вместо того чтобы каждый раз лезть в базу, храним частые ответы наготове.

**Cache hit / cache miss** — нашли в шпаргалке или нет.
Hit = ответ уже есть, отдаём мгновенно.
Miss = надо посчитать/достать и записать в шпаргалку на будущее.

**TTL (Time To Live)** — срок годности шпаргалки.
5 минут = через 5 минут шпаргалка "протухает" и нужно обновить.
Трейд-офф: короткий TTL = свежие данные, но больше miss.
          длинный TTL = быстро, но данные могут устареть.

**Latency** — задержка, время от запроса до ответа.
50ms → 2ms = в 25 раз быстрее для пользователя.

**Primary database** — главная база данных (PostgreSQL).
"Склад" — там всё хранится надёжно, но доступ медленнее.

**Зачем всё это:**
Причина: PostgreSQL не успевает обрабатывать все запросы
Следствие: Пользователи ждут меньше, сервер выдерживает больше нагрузки
Трейд-офф: +скорость, +масштабируемость, −сложность, −согласованность данных
```

## Vocabulary Builder (Термины + Аналогии)

| Термин | Аналогия | Зачем знать термин |
|--------|----------|-------------------|
| **API** (Application Programming Interface) | "Меню ресторана" — список того что можно заказать у системы | REST API, GraphQL API, "документация API" |
| **Database** | "Картотека" или "Excel на стероидах" | SQL database, NoSQL, in-memory DB |
| **Cache** (кэш) | "Шпаргалка" — готовый ответ чтобы не считать заново | cache hit/miss, cache invalidation, Redis cache |
| **Queue** (очередь) | "Очередь в кассу" — задачи ждут обработки по одной | message queue, job queue, FIFO |
| **Middleware** | "Охранник на входе" — проверяет/меняет всё что проходит | Express middleware, middleware chain |
| **Webhook** | "Обратный звонок" — "перезвоните мне когда будет готово" | webhook URL, webhook payload |
| **ORM** (Object-Relational Mapping) | "Переводчик" между объектами кода и таблицами БД | Prisma, Sequelize, "ORM vs raw SQL" |
| **Migration** (миграция) | "Ремонт без выселения" — меняем схему БД пока она работает | run migrations, rollback migration |
| **State** (состояние) | "Память" компонента между рендерами | stateful/stateless, state management |
| **Props** (пропсы) | "Инструкция от родителя" — данные сверху вниз | prop drilling, props validation |
| **Hook** (хук) | "Суперспособность" — дай функции память/эффекты | useState, useEffect, custom hooks |
| **Promise** (промис) | "Расписка" — обещание вернуть результат позже | Promise.all, async/await, pending/resolved/rejected |
| **Async** (асинхронный) | "Фоновая задача" — не блокирует остальное | async function, asynchronous I/O |
| **Type** (тип) | "Форма для печенья" — какой формы должны быть данные | type checking, TypeScript types |
| **Interface** (интерфейс) | "Контракт" — что объект обязуется уметь | implement interface, interface vs type |
| **Dependency** (зависимость) | "Ингредиент" — что нужно чтобы это работало | dependency injection (DI), dependency graph |
| **Build** (сборка) | "Из теста в хлеб" — из исходников в готовый продукт | build process, build artifacts |
| **Deploy** (деплой) | "Выкладка" — из твоего компа на сервер | deployment pipeline, deploy to production |
| **CI/CD** | "Конвейер" — автоматическая проверка и выкладка | CI pipeline, continuous deployment |

## Examples of Good Explanations

### Docker (контейнеризация)

```
Docker — это containerization («контейнеризация»), как грузовые контейнеры для кода.

Проблема: "У меня работает!" — "А у меня не работает!"
(термин: "works on my machine" — классическая боль разработки)

Решение: Упаковать приложение вместе со ВСЕМ что ему нужно в container (контейнер).
Как контейнер на корабле — неважно какой корабль, контейнер одинаковый.

Ключевые термины:
- Image (образ) — "рецепт" контейнера, read-only шаблон
- Container (контейнер) — запущенный экземпляр образа
- Dockerfile — инструкция как собрать образ
- Volume (том) — постоянное хранилище (данные не теряются при перезапуске)

Трейд-офф:
+ Portability — работает везде одинаково
+ Isolation — один контейнер не сломает другой
- Overhead — дополнительный слой (чуть медленнее)
- Learning curve — нужно учить новый инструмент
```

### Git (система контроля версий)

```
Git — это VCS (Version Control System, система контроля версий),
машина времени для кода с параллельными вселенными.

Ключевые термины:
- Commit (коммит) — snapshot, "фотография" кода в момент времени
- Branch (ветка) — параллельная вселенная для экспериментов
- Merge (мёрж) — соединение двух веток
- Pull Request / PR — "можно я вольюсь в main?" (запрос на слияние)
- Rebase (ребейз) — переписать историю, "пересадить" коммиты на другую базу

Почему не просто копировать папки?
- Delta storage — Git хранит только изменения (diff), экономия места
- Blame/history — можно точно увидеть кто и когда что изменил
- Concurrent work — несколько человек работают одновременно

Трейд-офф:
+ Full history — полная история всех изменений
+ Collaboration — совместная работа без конфликтов (почти)
- Mental model — сложная ментальная модель (DAG, directed acyclic graph)
- Merge conflicts — конфликты при слиянии веток
```

## Anti-Patterns

```
❌ Только термин (не учит):
"Это паттерн Observer из GoF"

❌ Только аналогия (прячет термин):
"Это как подписка на YouTube"

✅ Аналогия + термин (учит!):
"Это паттерн Observer («наблюдатель») — как подписка на YouTube:
 когда выходит новое видео, всем подписчикам приходит уведомление.
 (GoF = Gang of Four — классическая книга про паттерны, 1994)"
```

```
❌ "Используем eventual consistency"

✅ "Eventual consistency («согласованность в конечном счёте») —
   данные синхронизируются не мгновенно, как слухи в деревне:
   до всех дойдёт, но не сразу.

   Противоположность: strong consistency — все видят одно и то же мгновенно
   (как объявление по радио — все слышат одновременно)"
```

```
❌ "Это O(n log n) алгоритм"

✅ "Сложность O(n log n) — «эн лог эн»:
   Чем больше данных, тем дольше, но рост замедляется.
   1000 элементов ≈ 10,000 операций (не миллион!)

   Big O notation — способ описать как растёт время работы.
   O(n) = линейно, O(n²) = квадратично (плохо), O(log n) = логарифмически (круто)"
```

```
❌ "Реализуем через DI контейнер"

✅ "Dependency Injection (DI, «внедрение зависимостей») —
   вместо того чтобы компонент сам искал свои ингредиенты,
   мы их ему передаём снаружи.

   Как официант приносит меню, а не клиент идёт на кухню.

   DI Container — «коробка с ингредиентами» которая знает что кому нужно"
```

## Checklist

**Этап 1 — Факты:**
- [ ] Описано что произошло техническим языком (не упрощая)
- [ ] Все термины и жаргон на месте
- [ ] Хронология действий понятна

**Этап 2 — Понимание:**
- [ ] Каждый термин разобран с аналогией
- [ ] Дана этимология или перевод (откуда слово)
- [ ] Указаны связанные термины (где ещё встретишь)
- [ ] Объяснены причины (зачем это нужно)
- [ ] Показаны следствия (что это даёт)
- [ ] Указаны трейд-оффы (что получили / потеряли)

**Результат:**
- [ ] Можно пересказать другу без IT-бэкграунда
- [ ] Можно использовать термины в разговоре с разработчиком
- [ ] Понятно почему, а не только что
