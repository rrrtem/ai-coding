# Explain — Deep Understanding Mode

## Overview

Объясняй **в два этапа:**

1. **ЧТО ПРОИСХОДИТ** — факты, термины, жаргон как есть
2. **ЧТО ЭТО ЗНАЧИТ** — разбор каждого термина с аналогиями, причинами, следствиями

**Золотое правило:** Сначала реальность, потом понимание. Не упрощай факты — объясняй их.

**Почему термины важны:**
- Гуглить проблемы ("cache invalidation" → тысячи решений)
- Читать документацию и Stack Overflow
- Общаться с разработчиками
- Понимать что происходит в реальных системах

**Цель:** Видеть реальность как она есть + понимать её глубоко + говорить на профессиональном языке.

## Карточка объяснения

Единый шаблон для разбора любой концепции в Этапе 2. Масштабируется по сложности.

### Шаблон

```
**[Название]**
- Что: [название, категория, что делает -- техническим языком]
- Простыми словами: [одна яркая бытовая аналогия]
- Зачем: [какую проблему решает, причина -> следствие]
- Связи: [от чего зависит, что зависит от него, место в системе]
- Альтернативы: [что ещё есть, трейд-оффы выбора]
- Когда пригодится: [конкретный сценарий из практики]
```

### Инструкции по полям

**Что** -- название, категория, что делает
- Техническим языком, не упрощая. Термин на английском + перевод/транслитерация
- Указать категорию (runtime, package manager, design pattern, architectural approach...)
- Это "исходный код" объяснения -- реальность как она есть

**Простыми словами** -- аналогия из повседневной жизни
- Одна яркая аналогия, не три размытых
- Этимология если помогает ("cache от франц. cacher -- прятать")
- Аналогия должна быть точной: если она ломается при углублении -- заменить

**Зачем** -- какую проблему решает
- Причина: что было плохо без этого
- Следствие: что стало лучше с этим
- Формат "было -> стало" работает лучше всего

**Связи** -- место в экосистеме
- От чего зависит (prerequisites)
- Что зависит от него (dependents)
- На каком "этаже" абстракции живёт (физика -> код -> система -> пользователь)
- Связанные термины (где ещё встретишь это слово)

**Альтернативы** -- что ещё существует и почему выбрали это
- 1-2 альтернативы, не каталог
- Трейд-оффы выбора: что получили / что потеряли
- Не "X лучше Y", а "X лучше для Z, Y лучше для W"

**Когда пригодится** -- конкретный сценарий
- Реальная ситуация, не абстракция
- "Ты делаешь X, случается Y, и тогда это спасает" -- формат мини-истории

### Правило масштабирования

- **Крупная концепция** (инструмент, архитектурный паттерн, технология) -- все 6 полей
- **Средняя** (решение, подход, проблема) -- 3-4 ключевых поля по ситуации
- **Мелкая** (термин в проходящем объяснении) -- "Что" + "Простыми словами", остальное если действительно полезно

Карточка работает для любых концепций: инструменты, архитектурные решения, подходы, проблемы, технические решения.

## Explain Formula (Два этапа)

**Порядок критичен: сначала ФАКТ, потом ПОНИМАНИЕ.**

```
ЭТАП 1 — ЧТО ПРОИСХОДИТ (факты, термины, жаргон)
─────────────────────────────────────────────────
Описываем реальность как есть, техническим языком.
Не упрощаем, не прячем термины. Это "исходный код" объяснения.

ЭТАП 2 — ЧТО ЭТО ЗНАЧИТ (карточки объяснения)
─────────────────────────────────────────────────
Каждая концепция разбирается по Карточке объяснения.
Масштаб карточки зависит от сложности:
- Крупная концепция — все 6 полей
- Средняя — 3-4 поля
- Мелкая — "Что" + "Простыми словами"
```

### Пример двухэтапного объяснения

```
═══ ЭТАП 1: ЧТО ПРОИСХОДИТ ═══

Мы подняли Redis как кэш-слой перед PostgreSQL.
API сначала проверяет cache hit в Redis (TTL 5 минут),
при cache miss идёт в PostgreSQL и записывает результат в Redis.
Это уменьшает latency с ~50ms до ~2ms для горячих данных
и снижает нагрузку на primary database.

═══ ЭТАП 2: ЧТО ЭТО ЗНАЧИТ ═══

**Redis**
- Что: in-memory database (база данных в оперативной памяти)
- Простыми словами: блокнот на столе vs папка в шкафу --
  доступ мгновенный, но места мало
- Зачем: обычная БД хранит данные на диске, это надёжно но медленно.
  Redis держит всё в RAM -- в сотни раз быстрее
- Связи: работает перед PostgreSQL (основной БД), используется
  приложением через клиентскую библиотеку (ioredis, redis-py)
- Альтернативы: Memcached (проще, только key-value),
  встроенный кэш приложения (не расшарить между серверами)
- Когда пригодится: у тебя страница грузится 3 секунды,
  профилируешь -- 80% времени уходит на одни и те же запросы к БД.
  Ставишь Redis, эти запросы отдаются за миллисекунды

**TTL (Time To Live)**
- Что: срок жизни записи в кэше
- Простыми словами: срок годности на упаковке -- после него
  "продукт" выбрасывается и готовится заново
- Зачем: без TTL кэш показывал бы устаревшие данные вечно.
  Короткий TTL = свежие данные но больше промахов,
  длинный = быстро но данные могут устареть

**Cache hit / cache miss**
- Что: нашли ответ в кэше (hit) или нет (miss)
- Простыми словами: открыл шпаргалку -- ответ есть (hit),
  ответа нет -- придётся считать самому (miss)
```

Масштабирование в действии: Redis -- полная карточка (6 полей), TTL -- 3 поля, Cache hit/miss -- 2 поля.

## Examples of Good Explanations

### Docker (контейнеризация)

```
**Docker**
- Что: containerization platform (платформа контейнеризации) --
  упаковывает приложение со всем окружением в изолированный контейнер
- Простыми словами: грузовой контейнер для кода.
  Неважно какой корабль (сервер) -- контейнер одинаковый
- Зачем: "У меня работает!" -- "А у меня не работает!"
  (классическая боль: works on my machine).
  Docker гарантирует одинаковое поведение везде
- Связи: требует Docker Engine на хосте. Поверх Docker работают
  Kubernetes (оркестрация), Docker Compose (мульти-контейнерные приложения).
  Использует Linux namespaces и cgroups на уровне ядра
- Альтернативы: Podman (без демона, rootless по умолчанию),
  виртуальные машины (тяжелее, но полная изоляция ОС)
- Когда пригодится: новый разработчик приходит в проект.
  Вместо "поставь PostgreSQL 15, Redis 7, Node 20, настрой переменные..." --
  одна команда `docker compose up` и всё работает

Ключевые термины:
- Image (образ) -- "рецепт" контейнера, read-only шаблон
- Container (контейнер) -- запущенный экземпляр образа
- Dockerfile -- инструкция как собрать образ
- Volume (том) -- постоянное хранилище (данные не теряются при перезапуске)
```

### Git (система контроля версий)

```
**Git**
- Что: distributed version control system (распределённая система
  контроля версий) -- отслеживает все изменения в коде с полной историей
- Простыми словами: машина времени для кода с параллельными вселенными.
  Можно откатиться в любой момент прошлого, а ветки -- это
  параллельные реальности для экспериментов
- Зачем: без VCS -- копирование папок "проект_финал_v2_точно_финал".
  Git хранит только изменения (delta storage), знает кто и когда
  что менял (blame), позволяет работать нескольким людям одновременно
- Связи: Git -- фундамент. Поверх него: GitHub/GitLab (хостинг + код-ревью),
  CI/CD (автоматическая сборка при пуше), IDE-интеграции.
  Внутри -- DAG (directed acyclic graph) коммитов
- Альтернативы: Mercurial (проще ментальная модель, но проиграл в популярности),
  SVN (централизованный, проще но без оффлайн-работы)
- Когда пригодится: ты сломал что-то в коде час назад и не помнишь что именно.
  `git diff HEAD~5` покажет все изменения за последние 5 коммитов.
  Или `git bisect` найдёт точный коммит где сломалось

Ключевые термины:
- Commit (коммит) -- snapshot, "фотография" кода в момент времени
- Branch (ветка) -- параллельная вселенная для экспериментов
- Merge (мёрж) -- соединение двух веток
- Pull Request / PR -- "можно я вольюсь в main?" (запрос на слияние)
- Rebase (ребейз) -- переписать историю, "пересадить" коммиты на другую базу
```

## Anti-Patterns

```
❌ Только термин (не учит):
"Это паттерн Observer из GoF"

❌ Только аналогия (прячет термин):
"Это как подписка на YouTube"

✅ Аналогия + термин (учит!):
"Это паттерн Observer («наблюдатель») — как подписка на YouTube:
 когда выходит новое видео, всем подписчикам приходит уведомление.
 (GoF = Gang of Four — классическая книга про паттерны, 1994)"
```

```
❌ "Используем eventual consistency"

✅ "Eventual consistency («согласованность в конечном счёте») —
   данные синхронизируются не мгновенно, как слухи в деревне:
   до всех дойдёт, но не сразу.

   Противоположность: strong consistency — все видят одно и то же мгновенно
   (как объявление по радио — все слышат одновременно)"
```

```
❌ "Это O(n log n) алгоритм"

✅ "Сложность O(n log n) — «эн лог эн»:
   Чем больше данных, тем дольше, но рост замедляется.
   1000 элементов ≈ 10,000 операций (не миллион!)

   Big O notation — способ описать как растёт время работы.
   O(n) = линейно, O(n²) = квадратично (плохо), O(log n) = логарифмически (круто)"
```

```
❌ "Реализуем через DI контейнер"

✅ "Dependency Injection (DI, «внедрение зависимостей») —
   вместо того чтобы компонент сам искал свои ингредиенты,
   мы их ему передаём снаружи.

   Как официант приносит меню, а не клиент идёт на кухню.

   DI Container — «коробка с ингредиентами» которая знает что кому нужно"
```

## Checklist

**Этап 1 -- Факты:**
- [ ] Описано что произошло техническим языком (не упрощая)
- [ ] Все термины и жаргон на месте
- [ ] Хронология действий понятна

**Этап 2 -- Понимание (карточки):**
- [ ] Каждая концепция получила карточку подходящего масштаба
- [ ] Крупные концепции: все 6 полей
- [ ] "Простыми словами" -- точная аналогия, не размытая
- [ ] "Зачем" -- причина и следствие, не просто описание
- [ ] "Связи" -- понятно место в системе
- [ ] "Альтернативы" -- трейд-оффы, не "X лучше Y"
- [ ] "Когда пригодится" -- конкретный сценарий, не абстракция

**Результат:**
- [ ] Можно пересказать другу без IT-бэкграунда
- [ ] Можно использовать термины в разговоре с разработчиком
- [ ] Понятно почему, а не только что
