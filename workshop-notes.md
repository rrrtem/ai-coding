## 1. Git и модель версий

**Цепочка: папка → git → локал → remote → коммит → конфликты**

- Папка проекта — просто директория с файлами. Git ничего не знает о ней, пока не скажешь `git init`
- `git init` — превращает папку в репозиторий. Появляется скрытая `.git/`, в ней вся история
- Локальный репо — полная копия истории у тебя на машине. Можно работать без интернета
- Remote (GitHub/GitLab) — удалённая копия репо. Синхронизация, бэкап, совместная работа
- Локал vs remote — два независимых хранилища. `push` отправляет твои коммиты наверх, `pull` забирает чужие вниз
- Коммит — снимок состояния проекта в момент времени. Не "сохранение файла", а "сохранение всего проекта целиком"
- Коммит = что изменилось + кто + когда + зачем (сообщение). Атомарная единица истории
- Конфликт — два человека изменили одно и то же место. Git не знает чью версию оставить → просит решить руками
- Конфликт — не ошибка, а нормальная ситуация. Маркеры `<<<<<<<` / `=======` / `>>>>>>>` показывают обе версии
- Главное правило: коммить часто, пуш регулярно, пулл перед работой

## 2. Инструменты разработки = руки агента

**Агент без инструментов = мозг без рук. Умный, но беспомощный.**

- Claude Code по умолчанию: читает файлы, пишет файлы, выполняет команды в терминале, ищет по коду
- Bash — основной инструмент. Через него: установка пакетов, запуск серверов, тесты, git, деплой
- Read/Edit/Write — работа с файлами напрямую, без `cat` и `sed`. Точнее и безопаснее
- Glob/Grep — поиск файлов и кода. Агент находит нужное за секунды в проекте любого размера
- MCP-серверы — расширения. Подключают внешние системы: Figma, БД, браузер, API. Каждый MCP = новый набор "рук"
- Playwright — агент может открывать браузер, кликать, заполнять формы, делать скриншоты
- Субагенты — Claude запускает "помощников" для параллельных задач. Один ищет код, другой читает доки, третий пишет тесты
- Чем больше инструментов подключено, тем автономнее агент. Но каждый инструмент = поверхность атаки → безопасность

## 3. Vision-документ проекта

**Первый документ перед кодом. Что строим, для кого, зачем.**

- Vision = видение проекта. 1-2 страницы. Пишется до выбора технологий, до первой строки кода
- Процесс: агент задаёт вопросы по одному, пока не поймёт идею достаточно хорошо. Не выдумывает — спрашивает
- Что внутри: проблема → идея → сегменты ЦА → основной юзкейс → бизнес-логика → метрики успеха → амбиции
- Проблема — какую боль решаем. Если не можешь сформулировать проблему — рано писать код
- Сегменты ЦА — кто будет пользоваться. Конкретные люди с конкретными задачами, не "все"
- Основной юзкейс — один главный сценарий от начала до конца. Если он не работает — продукт не работает
- Метрики успеха — как поймёшь, что получилось. Числа, не ощущения
- Чего НЕТ в vision: технической реализации. Никаких фреймворков, баз, архитектур. Только "что" и "для кого"
- Зачем: без vision проект расползается. Каждое решение проверяешь: "это помогает нашему юзкейсу?"
- Демо: `/brainstorming` → агент ведёт диалог → на выходе готовый vision-документ

## 4. Техстек и обучение по ходу

**Какие технологии нужны для vision. Трейдоффы. Без оверинжиниринга.**

- Техстек выбирается ПОСЛЕ vision. Сначала "что строим", потом "из чего"
- Принцип: минимальный стек, который закрывает юзкейс. Не добавляй технологию "на всякий случай"
- Трейдоффы — у каждого выбора есть цена. Быстрее разработка = меньше контроля. Больше гибкости = больше сложности
- Оверинжиниринг — главный враг. Микросервисы для todo-приложения = пушка по воробьям
- Агент помогает выбрать: описываешь проект → получаешь рекомендацию с обоснованием → корректируешь
- `/explain` — скилл для обучения по ходу. Встретил незнакомую технологию → explain разбирает: что это, зачем, как работает в контексте
- Два этапа explain: сначала факты (что делает, входы/выходы), потом разбор (почему так, какие альтернативы)
- Цель: не просто использовать инструмент, а понимать почему именно он. Explain помогает учиться, а не просто копировать

## 5. Цикл итерации

**Спека → план → код → ревью → дебаг → коммит → пуш → деплой**

- Спека (spec) — описание что строим. Чем точнее спека, тем меньше переделок. Формат: цель + требования + ограничения
- План — как строим. Разбивка на шаги, порядок, зависимости. Агент предлагает план → ты корректируешь → агент выполняет
- Код — агент пишет, ты направляешь. Не "напиши мне приложение", а "реализуй первый шаг плана"
- Ревью — проверка написанного. Агент может ревьюить сам себя (через субагентов), но финальное слово за человеком
- Дебаг — что-то сломалось. Агент читает ошибку, ищет причину, предлагает фикс. Часто чинит сам
- Коммит — зафиксировал рабочее состояние. Одна логическая единица изменений = один коммит
- Пуш — отправил на remote. Теперь код в безопасности и доступен другим
- Деплой — опубликовал для пользователей. Vercel/Supabase/VPS — зависит от проекта
- Весь цикл повторяется. Каждая итерация — маленький инкремент. Не пытайся сделать всё за один проход

## Будущие темы

> Тезисами. Раскроются в следующих воркшопах.

### Безопасность и практики

- OWASP Top 10 — что ломают чаще всего
- Секреты: `.env`, переменные окружения, почему не коммитить пароли
- npm audit, ggshield, dependabot — автоматическая защита
- Принцип наименьших привилегий — давай минимум доступа

### UI/UX, дизайн

- Figma → код: workflow через MCP
- Компонентный подход: атомы → молекулы → организмы
- Адаптивность: mobile-first, breakpoints
- Доступность (a11y) — не опция, а требование

### Улучшение среды AI-разработки

- Claude Code skills — кастомные навыки под проект
- CLAUDE.md — как правильно писать инструкции для агента
- MCP-серверы — какие есть, как подключать, как писать свои
- Хуки — автоматические действия на события (pre-commit, post-save)
- Промпт-инженерия для кода — чем точнее запрос, тем точнее результат

### Концепты софтверной инженерии

- Compounding — маленькие улучшения накапливаются экспоненциально
- Компоненты — изолированные блоки с чётким интерфейсом
- Системный дизайн — как части связаны между собой
- Архитектурные принципы: SOLID, DRY, KISS — без фанатизма
- Технический долг — что это, когда допустимо, когда нет

### Backend: фреймворки и юзкейсы

- Next.js API Routes — когда хватает "бекенда внутри фронтенда"
- Express/Fastify — классический Node.js сервер
- FastAPI (Python) — ML-сервисы, быстрое прототипирование
- Supabase — backend-as-a-service: база + auth + storage из коробки
- Когда что выбирать — матрица решений по типу проекта

### Frontend: фреймворки и юзкейсы

- Next.js — полноценные веб-приложения, SSR, SEO
- React Native / Expo — мобильные приложения из одного кода
- Vite + React — SPA без SSR, когда SEO не важен
- Astro — контентные сайты, блоги, лендинги
- Telegram Mini Apps — фронтенд внутри мессенджера

### Базы данных

- Схема (schema) — структура данных до первой строки кода
- ORM (Prisma/Drizzle) — работа с БД через код, а не через SQL
- Миграции — версионирование структуры БД. Как git, но для таблиц
- Поля и типы — string, int, boolean, jsonb, uuid — когда что
- Связи — one-to-many, many-to-many. Как моделировать отношения
- Стек: PostgreSQL + Supabase + Prisma — рабочая комбинация для старта

### Тестирование и качество

- Зачем тесты — ловят баги до пользователей
- Уровни: unit → integration → e2e. Пирамида тестирования
- Vitest/Jest — для логики, Playwright — для UI
- TDD с AI — агент пишет тест → потом код → тест проходит

### DevOps и инфраструктура

- CI/CD — код пушнул → тесты прошли → деплой автоматом
- Docker — "у меня работает" больше не оправдание
- VPS vs serverless — когда свой сервер, когда облако
- Мониторинг — как узнать что приложение упало раньше пользователей
