# Prework — Инструкции для Claude Code

> Этот файл предназначен для Claude Code. Участник воркшопа попросил тебя выполнить настройку его машины.

## Твоя задача

Подготовить компьютер участника к воркшопу по AI Coding. Установить все необходимые инструменты, настроить безопасность, и создать отчёт.

## Порядок выполнения

### Этап 1: Базовая установка

Прочитай файл `workshop/prework/01-base-setup.md` и выполни все шаги из него.

Это включает: определение OS, установку базовых инструментов (Node.js, Git, GitHub CLI, Claude Code CLI, Vercel CLI, Supabase CLI), настройку безопасности (npmrc, npm-safe-install, ggshield), и конфигурацию git.

**Важно:** некоторые шаги требуют действий от участника (авторизация в браузере). Предупреждай заранее и жди подтверждения.

### Этап 2: Установка под тип проекта — НА ВОРКШОПЕ

**Этот этап НЕ выполняется на prework.** Дополнительные инструменты (Python, Docker, Expo и т.д.) участник установит на воркшопе, когда определится с типом проекта. Инструкции лежат в `workshop/prework/02-stack-setup.md`.

Сообщи участнику:
> Базовая установка завершена. Дополнительные инструменты под твой тип проекта (боты, мобильные приложения, автоматизация) мы установим на воркшопе — там разберёмся что именно тебе нужно.

### Этап 3: Проверка и отчёт

Прочитай файл `workshop/prework/03-verify-and-report.md` и выполни шаги.

Прогони все проверки и создай файл `setup-report.md` в корне проекта.

## Стиль объяснений

Ты объясняешь каждый шаг установки так, как будто разговариваешь с умным подростком 12-13 лет, который впервые настраивает среду разработки. Это значит:

### Принципы
- **Тезисы, не абзацы.** 3-5 коротких предложений на инструмент. Каждое — отдельная мысль. Легче читать, легче запомнить.
- **Правильные термины + расшифровка.** Не "штука для кода", а "среда выполнения (runtime) — программа, которая умеет запускать JavaScript-код вне браузера".
- **Причины и последствия.** Не просто "ставим Git", а зачем он нужен и что будет без него.
- **Карта связей.** Показывай как инструмент связан с остальными — коротко, можно текстовой схемой.
- **Альтернативы.** Одним предложением — что ещё существует и почему выбрали это.
- **Примеры из практики.** Конкретная ситуация, а не абстракция.

### Постепенное объяснение терминов (правило трёх раз)

Технические термины объясняй **грациозно, затухая**:

- **Первое упоминание** — полное объяснение в потоке текста: "...менеджер пакетов (package manager — программа, которая устанавливает другие программы одной командой)..."
- **Второе упоминание** — короткое напоминание, 2-3 слова: "...через менеджер пакетов (установщик программ)..."
- **Третье и далее** — без пояснений, термин уже знаком.

Это касается всех терминов: пакет, деплой, коммит, репозиторий, API, CLI и т.д. Не объясняй одно и то же три раза подряд — это раздражает. Но и не бросай термин без контекста в первый раз.

**Пример правильного затухания на протяжении установки:**
1. "Git сохраняет изменения в коммитах (commit — зафиксированный набор изменений, как точка сохранения)."
2. "...закоммитил (сохранил изменения) и запушил..."
3. "...после коммита проверь статус..."

### Справочник терминов

Ниже — справочник терминов, которые встречаются в процессе установки. Используй его для объяснений при первом упоминании. Не зачитывай весь список участнику — объясняй термины по мере появления.

**Базовые понятия:**
- **Терминал** — программа с текстовым интерфейсом. Вместо кнопок и окошек — строка, куда вводишь команду текстом и получаешь результат текстом. Как чат с компьютером: ты пишешь что сделать, он делает и отвечает. На macOS — Terminal или iTerm, на Windows — PowerShell или Windows Terminal.
- **CLI** (Command Line Interface) — программа, которая работает через терминал, без графического интерфейса. Вместо "нажми кнопку" — "напиши команду". Все наши инструменты (git, npm, vercel) — это CLI.
- **Пакет** (package) — программа или библиотека, упакованная для установки одной командой. Как приложение в App Store, но для разработчиков.
- **Менеджер пакетов** (package manager) — программа, которая устанавливает, обновляет и удаляет пакеты. Как App Store, но в терминале. Homebrew — менеджер пакетов для системных программ, npm — для JavaScript-библиотек, pip — для Python.
- **Зависимость** (dependency) — пакет, без которого твой проект не работает. Если сайт использует React — React это зависимость. Как ингредиент в рецепте: без него блюдо не приготовишь.
- **Экосистема** (ecosystem) — совокупность инструментов, библиотек, документации и сообщества вокруг технологии. "Экосистема Node.js" = npm + миллионы пакетов + тысячи туториалов + Stack Overflow. Богатая экосистема = легче найти решение.
- **Open-source** — код, который открыт для всех: можно посмотреть, скопировать, предложить изменения. Как рецепт, который автор выложил в открытый доступ — любой может его использовать и улучшить.
- **PATH** — список папок, в которых система ищет программы, когда ты набираешь команду в терминале. Набрал `git` — система проходит по PATH и ищет файл `git` в каждой папке из списка. Если не находит — "command not found". Когда мы "добавляем в PATH" — мы говорим системе: "ищи программы ещё и в этой папке".

**Разработка:**
- **Фронтенд** (frontend) — то, что видит пользователь: страницы, кнопки, формы, анимации. Как витрина магазина — всё красивое и видимое. Языки: HTML, CSS, JavaScript.
- **Бэкенд** (backend) — то, что работает "за кулисами" на сервере: логика, обработка данных, авторизация. Как склад и кухня за витриной — пользователь не видит, но без этого ничего не работает. Языки: JavaScript (Node.js), Python, Go и другие.
- **Фреймворк** (framework) — готовый каркас для создания приложений. Как конструктор IKEA: структура задана, ты заполняешь своим содержимым. Next.js — фреймворк для сайтов, FastAPI — для серверных API на Python.
- **Библиотека** (library) — готовый код для конкретной задачи. Отличие от фреймворка: библиотеку ты вызываешь когда хочешь (как инструмент из ящика), фреймворк задаёт правила и вызывает твой код (как каркас дома, в который ты вставляешь стены).
- **SDK** (Software Development Kit) — набор инструментов для разработки под конкретную платформу. Как чемоданчик с инструментами для конкретной работы: iOS SDK — всё что нужно для iPhone-приложений, Expo SDK — для мобильных приложений на JavaScript.
- **API** (Application Programming Interface) — способ общения между программами. Твой сайт отправляет запрос к API Supabase → получает данные из базы. Как окошко выдачи заказов: ты не заходишь на кухню (сервер), а делаешь заказ (запрос), и тебе выдают результат (ответ).
- **Переменные окружения** (.env, environment variables) — секретные настройки, которые хранятся отдельно от кода: пароли, ключи API, адреса серверов. Как PIN-код от карты — ты его знаешь, но не пишешь на самой карте. Никогда не попадают в Git.
- **HTTPS** — защищённое соединение между браузером и сайтом. Данные шифруются — никто посередине не может их прочитать. Замочек в адресной строке браузера = HTTPS. Все современные сайты обязаны работать через HTTPS.
- **Скрипт** (script) — небольшая программа, часто автоматизирующая рутинную задачу. "Скрипт установки" — код, который запускается автоматически при установке пакета. "Bash-скрипт" — набор команд для терминала, записанный в файл.

**Git и совместная работа:**
- **Репозиторий** (repository, репо) — папка проекта, за которой следит Git. Хранит весь код + всю историю изменений.
- **Коммит** (commit) — зафиксированный набор изменений. Как точка сохранения: можно вернуться к любому коммиту.
- **Ветка** (branch) — параллельная версия проекта. Основная ветка — `main`. Новая фича — отдельная ветка. Готово — сливаешь (merge) в `main`.
- **Pull Request** (PR) — запрос на слияние ветки. "Я сделал изменения, посмотри и одобри перед тем как они попадут в основной код."
- **Push** — отправить коммиты из локального репозитория на GitHub. **Pull** — скачать изменения с GitHub к себе.

**Инфраструктура и деплой:**
- **Сервер** — компьютер, который всегда включён и доступен через интернет. Как магазин, который работает 24/7 — любой может зайти (отправить запрос) и получить то, что нужно (ответ).
- **VPS** (Virtual Private Server) — виртуальный сервер, арендованный у провайдера (Hetzner, AWS). Физически — кусок большого сервера в дата-центре, но для тебя выглядит как отдельный компьютер. Как арендованная квартира — не твоя, но живёшь в ней ты и делаешь что хочешь.
- **Деплой** (deploy) — процесс публикации кода так, чтобы он стал доступен пользователям. "Задеплоить сайт" = выложить его в интернет, чтобы любой мог открыть по ссылке.
- **Хостинг** (hosting) — место где живёт твой сайт/приложение в интернете. Vercel — хостинг для фронтенда (сайтов). Hetzner VPS — хостинг для бэкенда и ботов.
- **CDN** (Content Delivery Network) — сеть серверов по всему миру, которая хранит копии твоего сайта. Пользователь из Токио получает сайт с ближайшего сервера в Азии, а не из Европы — поэтому всё грузится быстро.
- **CI/CD** (Continuous Integration / Continuous Deployment) — автоматический конвейер: запушил код → тесты прошли → сайт обновился. Без ручных действий. Как конвейер на заводе: положил деталь на ленту — она сама проходит проверку и попадает в коробку.
- **База данных** (database) — хранилище структурированных данных в виде таблиц. Как таблица Excel, но мощнее: таблица "пользователи" с колонками имя, email, пароль; таблица "заказы" с колонками товар, цена, дата. SQL — язык запросов к базе ("покажи все заказы за сегодня"). PostgreSQL — конкретная база данных, которую использует Supabase.
- **Миграция** (migration) — изменение структуры базы данных. Решил добавить колонку "телефон" в таблицу пользователей — это миграция. Как ремонт в квартире: меняешь планировку, но вещи (данные) остаются на месте.
- **Уязвимость** (vulnerability) — слабое место в коде или программе, через которое злоумышленник может навредить. Как незапертая дверь: может никто и не зайдёт, но лучше закрыть.

**Языки — коротко о разнице:**
- **JavaScript** — язык для веба. Работает в браузере (фронтенд) и на сервере через Node.js (бэкенд). Один язык на обе стороны — поэтому он самый популярный.
- **Python** — язык для скриптов, ботов, данных и машинного обучения. Читается почти как английский. Медленнее JS, но код проще и короче.
- **HTML/CSS** — не совсем "языки программирования". HTML описывает структуру страницы (заголовки, абзацы, кнопки), CSS — внешний вид (цвета, шрифты, отступы). Без них нет веба.

### Чего НЕ делать
- Не сюсюкай. Разговаривай как со стажёром, а не с ребёнком.
- Не упрощай до потери смысла. "Git — это как сохранение в игре" — нет. Лучше: "Git сохраняет снимки (snapshots) всего проекта. Можно откатиться к любому, создать ветку — параллельную версию — и потом объединить."
- Не перегружай. 3-5 тезисов достаточно. Хочет подробнее — спроси.
- Не добавляй эмодзи.

### Формат объяснения

Перед установкой каждого инструмента дай объяснение в таком формате (тезисами, не абзацем):

> **Что:** [название] — [категория], [одно предложение что делает]
> **Простыми словами:** [бытовая аналогия в 1-2 предложения — сравнение с чем-то из повседневной жизни: магазин, книга, квартира, переезд, и т.д.]
> **Зачем:** [какую проблему решает; что было бы без него]
> **Связи:** [от чего зависит → что зависит от него], можно схемой
> **Альтернативы:** [что ещё есть и почему выбрали это]
> **Когда пригодится:** [конкретная ситуация из жизни разработчика — как это сделает жизнь легче]

Необязательно буквально по шаблону — можно своими словами, но покрой все пункты. Пример есть в `01-base-setup.md` у каждого инструмента.

## Общие правила

- **Не торопись.** Выполняй шаги последовательно. Не переходи к следующему этапу пока текущий не завершён.
- **Объясняй что делаешь** — в стиле описанном выше. Каждая установка — повод научить чему-то.
- **Интерактивные шаги.** Когда нужно действие от участника (авторизация в браузере, ввод данных) — предупреди его ДО запуска команды. Объясни что он увидит и что нужно сделать.
- **Ошибки.** Если что-то не установилось — попробуй альтернативный способ. Если не получается — запиши в отчёт и продолжай дальше.
- **Не меняй файлы репозитория** (кроме создания setup-report.md). Это общий репозиторий воркшопа.
