# Этап 1: Базовая установка

> Инструкции для Claude Code. Выполняй шаги последовательно.

## Принцип идемпотентности

**Каждый шаг БЕЗОПАСЕН для повторного запуска.** Участник мог уже что-то установить раньше. Порядок действий для каждого инструмента:

1. **Проверь** — установлен ли уже? (`command -v <tool>`, `<tool> --version`)
2. **Если установлен** — сообщи версию и **пропусти** установку. Не обновляй без явной необходимости.
3. **Если установлен, но версия слишком старая** — предупреди участника и предложи обновить. Не обновляй молча.
4. **Если не установлен** — установи и проверь.

**Для конфигов** (~/.npmrc, git config, PATH):
- Проверь текущее значение перед изменением
- Если конфиг уже содержит нужные строки — не дублируй
- Если участник уже настроил git user.name/email — не перезаписывай, просто покажи текущие значения и спроси "оставить так?"

**Никогда не делай:**
- `brew upgrade` без спроса (может сломать другие проекты)
- Перезаписывание существующих конфигов целиком
- Удаление существующих PATH записей

## 1.1 Определи операционную систему

Определи OS участника (`uname -s` или аналог). Дальнейшие команды зависят от OS:
- **macOS** — основной путь (Homebrew)
- **Linux** — apt/dnf/pacman в зависимости от дистрибутива
- **Windows (WSL)** — apt внутри WSL

Сообщи участнику какую OS ты определил.

## 1.2 Менеджер пакетов

### macOS: Homebrew

Проверь: `brew --version`

Если не установлен — объясни участнику:

> **Что:** Homebrew — менеджер пакетов (package manager) для macOS. Программа, которая устанавливает другие программы одной командой в терминале.
> **Простыми словами:** App Store, но для инструментов разработчика. Только вместо кнопки "Установить" — команда `brew install`.
> **Зачем:** Без него каждый инструмент пришлось бы искать на сайте, скачивать установочный файл, устанавливать вручную, иногда ещё добавлять программу в PATH (список папок, где система ищет программы). Homebrew делает всё это за тебя: `brew install git` — и готово.
> **Связи:** Homebrew — фундамент. Через него мы поставим Git, Node.js, GitHub CLI, Supabase CLI и другие инструменты. Если его нет — каждую следующую установку придётся делать вручную.
> **Альтернативы:** MacPorts, Nix — тоже менеджеры пакетов для macOS. Homebrew выбран потому что у него самое большое сообщество и почти все туториалы в интернете написаны под него.
> **Когда пригодится:** Тебе нужно поставить 8 инструментов. Без Homebrew — это 8 сайтов, 8 скачиваний, 8 установок. С Homebrew — 8 команд `brew install`, каждая занимает секунды.

Установи (эта команда скачивает и запускает официальный скрипт установки Homebrew; `curl` — встроенная утилита для скачивания файлов из интернета через терминал):
```bash
/bin/bash -c "$(curl -fsSL https://raw.githubusercontent.com/Homebrew/install/HEAD/install.sh)"
```

**Важно:** после установки Homebrew на Apple Silicon (M1/M2/M3/M4 — новые процессоры Apple) может потребоваться добавить его в PATH (список папок, где система ищет программы):
```bash
echo 'eval "$(/opt/homebrew/bin/brew shellenv)"' >> ~/.zprofile
eval "$(/opt/homebrew/bin/brew shellenv)"
```

Проверь что работает: `brew --version`

### Linux: apt (Ubuntu/Debian)

Объясни участнику:

> **Что:** apt (Advanced Package Tool) — менеджер пакетов, встроенный в Ubuntu и Debian. Аналог Homebrew, но для Linux и уже предустановлен.
> **Простыми словами:** Тот же "App Store для разработчиков", но уже встроен в систему — ничего ставить не надо, сразу работает.
> **Зачем:** Устанавливает программы одной командой. `sudo apt install git` — и Git на месте. `sudo apt update` обновляет список доступных пакетов (как обновить каталог магазина перед покупкой — вдруг появилось что-то новое).
> **Связи:** apt — фундамент на Linux. Через него ставим Git, Python, и системные зависимости. Node.js ставится через отдельный репозиторий NodeSource, потому что в стандартном репозитории apt обычно лежит устаревшая версия.
> **Альтернативы:** dnf (Fedora), pacman (Arch Linux), snap, flatpak. apt выбран потому что Ubuntu/Debian — самые популярные дистрибутивы для серверов и разработки.

Проверь что `apt` доступен. Обнови индекс:
```bash
sudo apt update
```

## 1.3 Node.js 22+ (с npm 11+)

Проверь: `node -v` и `npm -v`

Нужно: Node.js 22+ и npm 11.x.x.

Если установлен и версии подходят — пропусти, сообщи текущие версии.

Если не установлен или версия слишком старая — объясни:

> **Что:** Node.js — среда выполнения (runtime) для JavaScript. Браузер умеет запускать JS только внутри веб-страниц. Node.js позволяет запускать его в терминале, на сервере — где угодно.
> **Простыми словами:** Если JavaScript — это бензин, то Node.js — двигатель, который может работать не только в машине (браузере), но и в генераторе, лодке, где угодно.
> **Зачем:** Почти все инструменты для веб-разработки написаны на JavaScript и образуют целую экосистему (ecosystem — инструменты + библиотеки + документация + сообщество вокруг технологии). Без Node.js ничего из этого не запустится.
> **Связи:** Node.js → npm (идёт в комплекте, менеджер пакетов) → Vercel CLI, Supabase CLI, и все библиотеки проекта. Node.js — корень дерева, без него ветки не работают.
> **Альтернативы:** Deno, Bun — тоже запускают JavaScript. Но 95% экосистемы заточено под Node.js. Нам нужна версия 22+, потому что с ней идёт npm 11 с улучшенной безопасностью.
> **Когда пригодится:** Ты хочешь задеплоить (выложить в интернет) сайт командой `vercel deploy`. Эта команда — JavaScript-программа. Без Node.js терминал скажет "command not found".

**macOS:**
```bash
brew install node
```

**Linux:**
```bash
curl -fsSL https://deb.nodesource.com/setup_22.x | sudo -E bash -
sudo apt install -y nodejs
```

Если Node.js установлен но старый (и npm ниже 11) — предложи обновить:
> У тебя Node.js [версия], npm [версия]. Для воркшопа нужен npm 11+. Обновить? Это может повлиять на другие проекты.

Обнови только после подтверждения:
```bash
brew upgrade node  # macOS
```

Проверь: `node -v` (должно быть v22+) и `npm -v` (должно быть 11.x.x)

## 1.4 Git

Проверь: `git --version`

Если установлен — пропусти установку, переходи к настройке.

Если не установлен — объясни:

> **Что:** Git — система контроля версий (version control system). Она отслеживает каждое изменение в каждом файле проекта.
> **Простыми словами:** "История правок" для всего проекта, как в Google Docs — только мощнее. Видно кто, когда и что менял. Можно откатить любое изменение.
> **Зачем:** Без Git ты не можешь: откатить код назад если что-то сломал, работать над проектом с другими людьми, хранить код на GitHub. Представь — ты добавил новую фичу и всё перестало работать. Без Git — сидишь и вспоминаешь что менял. С Git — одна команда `git diff`, и видишь все изменённые строчки.
> **Связи:** Git → GitHub (облачное хранилище для git-репозиториев) → GitHub CLI (управление GitHub из терминала) → Vercel (автоматически деплоит сайт при каждом `git push`). Без Git вся цепочка разработки разваливается.
> **Альтернативы:** Mercurial, SVN — старые системы контроля версий. Git победил: его используют 95%+ разработчиков, GitHub и все системы автоматической сборки/деплоя построены вокруг него.
> **Когда пригодится:** Ты делаешь сайт. Создаёшь ветку (branch) "новый-дизайн", экспериментируешь. Не понравилось — удалил ветку. Понравилось — слил (merge) в основную версию. Основной код всё время в безопасности.

**macOS:**
```bash
brew install git
```

**Linux:**
```bash
sudo apt install -y git
```

### Настройка git

Сначала проверь текущие настройки:
```bash
git config --global user.name
git config --global user.email
```

**Если уже настроено** — покажи участнику текущие значения и спроси:
> Git уже настроен: имя "[текущее имя]", email "[текущий email]". Оставить так?

**Если не настроено** — объясни и спроси:
> Git подписывает каждое изменение именем и email автора — чтобы было видно, кто что написал. Это как подпись под каждым коммитом (commit — зафиксированный набор изменений). Какое имя и email использовать? Они будут видны в истории проекта на GitHub.

После получения ответа:
```bash
git config --global user.name "Имя Участника"
git config --global user.email "email@example.com"
```

## 1.5 GitHub CLI

Проверь: `gh --version`

Если не установлен — объясни:

> **Что:** GitHub CLI (`gh`) — официальный инструмент для управления GitHub из терминала (CLI — программа, которая работает через текстовые команды, без окошек). GitHub — это облачная платформа где хранится код.
> **Простыми словами:** GitHub — это Google Drive для кода: хранит файлы в облаке, показывает историю изменений, позволяет работать вместе. GitHub CLI — пульт управления этим "Google Drive" прямо из терминала.
> **Зачем:** Без него для создания репозитория (репо — папка проекта под контролем Git), Pull Request-а (PR — запрос "посмотри мои изменения и одобри") или просмотра issues (задачи и баги проекта) нужно каждый раз открывать браузер. С `gh` — всё в терминале: `gh repo create`, `gh pr create`, `gh issue list`. Быстрее и удобнее.
> **Связи:** Git (отслеживает изменения локально) → GitHub (хранит код в облаке) → GitHub CLI (управляет GitHub из терминала). Git без GitHub работает, но код остаётся только на твоём компьютере. GitHub CLI без Git бесполезен.
> **Альтернативы:** Можно делать всё через сайт github.com вручную. Но на воркшопе мы будем часто создавать репозитории и PR — через CLI это в разы быстрее.
> **Когда пригодится:** Ты написал код, закоммитил, запушил. Теперь нужен Pull Request. Без CLI: открываешь браузер → github.com → находишь репо → жмёшь кнопки. С CLI: `gh pr create --title "Новая фича"` — готово за 3 секунды.

**macOS:**
```bash
brew install gh
```

**Linux:**
```bash
(type -p wget >/dev/null || sudo apt install wget -y) \
  && sudo mkdir -p -m 755 /etc/apt/keyrings \
  && out=$(mktemp) && wget -nv -O "$out" https://cli.github.com/packages/githubcli-archive-keyring.gpg \
  && cat $out | sudo tee /etc/apt/keyrings/githubcli-archive-keyring.gpg > /dev/null \
  && sudo chmod go+r /etc/apt/keyrings/githubcli-archive-keyring.gpg \
  && echo "deb [arch=$(dpkg --print-architecture) signed-by=/etc/apt/keyrings/githubcli-archive-keyring.gpg] https://cli.github.com/packages stable main" | sudo tee /etc/apt/sources.list.d/github-cli.list > /dev/null \
  && sudo apt update \
  && sudo apt install gh -y
```

### Авторизация GitHub CLI

Сначала проверь: `gh auth status`

**Если уже авторизован** — сообщи и пропусти:
> GitHub CLI уже авторизован как [username]. Отлично, пропускаю.

**Если не авторизован:**

**Предупреди участника ПЕРЕД запуском:**

> Сейчас нужно авторизовать GitHub CLI. Откроется браузер с GitHub — нужно будет нажать "Authorize". Готов?

Дождись подтверждения, затем попроси участника выполнить эту команду самостоятельно (она интерактивная):
```

```

Подскажи: выбрать GitHub.com → HTTPS → Login with a web browser.

Проверь: `gh auth status` — должно показать "Logged in to github.com"

## 1.6 Vercel CLI

Проверь: `vercel --version`

Если не установлен — объясни:

> **Что:** Vercel CLI — инструмент для управления платформой Vercel из терминала. Vercel — это хостинг (hosting) — место, где твой сайт живёт в интернете и доступен по ссылке.
> **Простыми словами:** Ты сделал сайт на своём компьютере — но пока он как рукопись в ящике стола, никто его не видит. Vercel — это типография + доставка: берёт твой код и делает его доступным для всего мира по ссылке.
> **Зачем:** Без хостинга сайт существует только на твоём компьютере. Vercel берёт код из GitHub и автоматически превращает его в работающий сайт с HTTPS (защищённое соединение — замочек в адресной строке), красивым доменом, и CDN (сеть серверов по всему миру, чтобы сайт быстро грузился из любой страны). Каждый `git push` — и сайт обновляется сам.
> **Связи:** Ты пишешь код → Git фиксирует изменения → `git push` отправляет на GitHub → Vercel подхватывает и деплоит. Vercel CLI позволяет управлять этим процессом: `vercel deploy`, `vercel env pull`, `vercel logs`.
> **Альтернативы:** Netlify, Cloudflare Pages, GitHub Pages — тоже хостинги. Vercel выбран потому что он создан авторами Next.js и лучше всего работает с React/Next.js проектами, которые мы используем на воркшопе.
> **Когда пригодится:** Ты сделал лендинг. Команда `vercel` — и через 30 секунд получаешь URL вроде `my-site.vercel.app`, который можно отправить кому угодно. Обновил код, запушил в GitHub — сайт обновился сам, без единой команды.

```bash
npm install -g vercel
```

Проверь: `vercel --version`

## 1.7 Supabase CLI

Проверь: `supabase --version`

Если не установлен — объясни:

> **Что:** Supabase CLI — инструмент для управления Supabase из терминала. Supabase — это готовый бэкенд "из коробки": база данных (хранилище данных в виде таблиц, как мощный Excel) + авторизация (логин/регистрация пользователей) + хранилище файлов.
> **Простыми словами:** Представь что ты открываешь магазин. Витрина (сайт) — это Vercel. А Supabase — это всё что за витриной: склад товаров (база данных), касса (авторизация), система учёта (API). Без этого магазин — просто красивая картинка.
> **Зачем:** Любое приложение с данными (пользователи, заказы, посты) нуждается в базе данных и API (способ общения между сайтом и сервером — как окошко выдачи заказов). Без Supabase пришлось бы самому поднимать сервер, ставить PostgreSQL, писать API, настраивать авторизацию — на это уходят дни. Supabase даёт всё это за минуты.
> **Связи:** Vercel (хостит фронтенд — витрину, то что видит пользователь) ↔ Supabase (хранит данные — склад за кулисами). Вместе они покрывают 80% потребностей типичного веб-приложения. CLI нужен для миграций (migration — изменение структуры таблиц в базе данных, например "добавить колонку телефон") и локальной разработки.
> **Альтернативы:** Firebase (Google), PlanetScale, Neon. Supabase выбран потому что он open-source (код открыт — можно посмотреть и убедиться что всё честно), использует PostgreSQL (самая популярная база данных), и имеет щедрый бесплатный план.
> **Когда пригодится:** Ты делаешь сайт с формой обратной связи. Supabase хранит все заявки в таблице. Ты видишь их в красивой панели управления, можешь фильтровать, экспортировать. Без Supabase — пришлось бы поднимать свой сервер, ставить PostgreSQL, писать API — на это уходят дни.

**macOS:**
```bash
brew install supabase/tap/supabase
```

**Linux:**
```bash
brew install supabase/tap/supabase
```
(Если Homebrew нет на Linux — установи через бинарник: скачай последний релиз с [github.com/supabase/cli](https://github.com/supabase/cli/releases))

Проверь: `supabase --version`

## 1.8 Глобальная безопасность: ~/.npmrc

Сначала проверь существует ли файл:
```bash
cat ~/.npmrc 2>/dev/null || echo "НЕ НАЙДЕН"
```

**Если файл существует** — проверь, содержит ли он нужные строки. Добавь только недостающие. НЕ перезаписывай файл целиком — участник мог добавить свои настройки.

**Если файла нет** — создай и объясни:

> **Что:** `.npmrc` — файл конфигурации npm. Лежит в домашней папке (`~`) и влияет на поведение npm во всех проектах.
> **Простыми словами:** Как настройки безопасности на телефоне: "не устанавливать приложения из неизвестных источников", "проверять на вирусы". Один раз включил — и забыл, оно защищает тебя в фоне.
> **Зачем:** npm по умолчанию доверяет всем пакетам — скачал и сразу запускает скрипты (scripts — небольшие программы, которые автор пакета заложил для автоматического запуска при установке). Злоумышленники создают пакеты с похожими названиями (`lodas` вместо `lodash`) и вшивают вредоносный код в эти скрипты. `.npmrc` меняет настройки по умолчанию на безопасные.
> **Связи:** npm (менеджер пакетов) читает `.npmrc` при каждой операции. Это влияет на установку всех пакетов во всех проектах — Vercel CLI, Supabase CLI, библиотеки проекта. Настроил один раз — работает везде.
> **Альтернативы:** Можно каждый раз добавлять флаги вручную (`npm install --ignore-scripts`), но это легко забыть. Файл `.npmrc` делает безопасное поведение дефолтным.
> **Когда пригодится:** Ты набрал `npm install reatc` (опечатка). Без `.npmrc` — npm скачает поддельный пакет и запустит его скрипты. С `ignore-scripts=true` — скрипты заблокированы, даже если пакет вредоносный, он не запустит свой код.

Нужные строки:
```ini
audit=true
ignore-scripts=true
engine-strict=true
fund=false
```

Для добавления недостающих строк:
```bash
grep -q "^audit=true" ~/.npmrc 2>/dev/null || echo "audit=true" >> ~/.npmrc
grep -q "^ignore-scripts=true" ~/.npmrc 2>/dev/null || echo "ignore-scripts=true" >> ~/.npmrc
grep -q "^engine-strict=true" ~/.npmrc 2>/dev/null || echo "engine-strict=true" >> ~/.npmrc
grep -q "^fund=false" ~/.npmrc 2>/dev/null || echo "fund=false" >> ~/.npmrc
```

Объясни участнику что означает каждая строка:
> - `audit=true` — при каждом `npm install` npm проверяет все пакеты по базе известных уязвимостей (vulnerability — слабое место в коде, через которое могут навредить, как незакрытая дверь). Если найдёт проблему — предупредит.
> - `ignore-scripts=true` — **самая важная.** Блокирует автоматический запуск скриптов при установке пакетов. Именно через такие скрипты вредоносные пакеты запускают свой код. Если легитимному пакету нужны скрипты (редко) — ты разрешишь вручную.
> - `engine-strict=true` — не даёт установить пакет, который не поддерживает твою версию Node.js. Спасает от загадочных ошибок "работает у всех, а у меня нет".
> - `fund=false` — убирает сообщения "этот пакет ищет спонсоров" при установке. Не влияет на безопасность, просто убирает шум.

## 1.9 Скрипт npm-safe-install

Объясни участнику:

> **Что:** `npm-safe-install` — наш собственный скрипт-обёртка вокруг `npm install`. Он проверяет пакет перед установкой: возраст, количество скачиваний, дату последней версии.
> **Простыми словами:** Как проверка отзывов перед покупкой на маркетплейсе. Товар появился вчера, 3 отзыва, продавец без рейтинга? Скорее всего, подделка. Этот скрипт делает такую проверку автоматически для каждого пакета.
> **Зачем:** `.npmrc` защищает от вредоносных скриптов, но не мешает установить подозрительный пакет. Этот скрипт добавляет второй уровень — предупреждает, если пакет выглядит подозрительно (слишком новый, мало скачиваний, свежая версия).
> **Связи:** `.npmrc` (пассивная защита — блокирует скрипты) + `npm-safe-install` (активная проверка — анализирует пакет) + `ggshield` (ловит секреты) = три уровня безопасности. Каждый закрывает свою "дыру".
> **Альтернативы:** `socket.dev`, `snyk` — коммерческие инструменты для проверки пакетов. Наш скрипт проще, но бесплатный и покрывает основные сценарии.
> **Когда пригодится:** Ты нашёл в интернете пакет `super-cool-parser`. Запускаешь `npm-safe-install super-cool-parser` — и видишь: "создан 3 дня назад, 12 скачиваний". Это красные флаги — скорее всего, это не тот пакет, который тебе нужен.

Создай файл `~/bin/npm-safe-install`:

```bash
#!/usr/bin/env bash
set -euo pipefail

# npm-safe-install — проверяет пакет перед установкой
# Использование: npm-safe-install <package-name> [--force]

RED='\033[0;31m'
GREEN='\033[0;32m'
YELLOW='\033[1;33m'
NC='\033[0m'

if [ $# -lt 1 ]; then
  echo "Usage: npm-safe-install <package-name> [--force]"
  exit 1
fi

PACKAGE="$1"
FORCE="${2:-}"

echo "Checking package: $PACKAGE"
echo "---"

# Fetch package info from registry
REGISTRY_DATA=$(curl -s "https://registry.npmjs.org/$PACKAGE" 2>/dev/null)

if echo "$REGISTRY_DATA" | grep -q '"error"'; then
  echo -e "${RED}ERROR: Package '$PACKAGE' not found on npmjs.com${NC}"
  exit 1
fi

# Get creation date
CREATED=$(echo "$REGISTRY_DATA" | node -e "let d='';process.stdin.on('data',c=>d+=c);process.stdin.on('end',()=>{try{console.log(JSON.parse(d).time.created)}catch{console.log('unknown')}})" 2>/dev/null || echo "unknown")

# Get latest version
LATEST=$(echo "$REGISTRY_DATA" | node -e "let d='';process.stdin.on('data',c=>d+=c);process.stdin.on('end',()=>{try{console.log(JSON.parse(d)['dist-tags'].latest)}catch{console.log('unknown')}})" 2>/dev/null || echo "unknown")

# Get latest version publish date
LATEST_DATE=$(echo "$REGISTRY_DATA" | node -e "let d='';process.stdin.on('data',c=>d+=c);process.stdin.on('end',()=>{try{const j=JSON.parse(d);console.log(j.time[j['dist-tags'].latest])}catch{console.log('unknown')}})" 2>/dev/null || echo "unknown")

# Get weekly downloads
DOWNLOADS=$(curl -s "https://api.npmjs.org/downloads/point/last-week/$PACKAGE" 2>/dev/null | node -e "let d='';process.stdin.on('data',c=>d+=c);process.stdin.on('end',()=>{try{console.log(JSON.parse(d).downloads||0)}catch{console.log(0)}})" 2>/dev/null || echo "0")

WARNINGS=0

# Helper: convert ISO date to epoch (works on both macOS and Linux)
to_epoch() {
  local dt="${1%%.*}"  # strip fractional seconds
  dt="${dt%%Z}"        # strip trailing Z
  date -j -f "%Y-%m-%dT%H:%M:%S" "$dt" "+%s" 2>/dev/null \
    || date -d "$dt" "+%s" 2>/dev/null \
    || node -e "console.log(Math.floor(new Date('$1').getTime()/1000))" 2>/dev/null \
    || echo "0"
}

# Check age > 365 days
if [ "$CREATED" != "unknown" ]; then
  CREATED_TS=$(to_epoch "$CREATED")
  NOW_TS=$(date "+%s")
  AGE_DAYS=$(( (NOW_TS - CREATED_TS) / 86400 ))
  if [ "$AGE_DAYS" -lt 365 ]; then
    echo -e "${YELLOW}WARNING: Package is only $AGE_DAYS days old (threshold: 365)${NC}"
    WARNINGS=$((WARNINGS + 1))
  else
    echo -e "${GREEN}OK: Package age: $AGE_DAYS days${NC}"
  fi
fi

# Check latest version age > 7 days
if [ "$LATEST_DATE" != "unknown" ]; then
  LATEST_TS=$(to_epoch "$LATEST_DATE")
  NOW_TS=$(date "+%s")
  VER_AGE_DAYS=$(( (NOW_TS - LATEST_TS) / 86400 ))
  if [ "$VER_AGE_DAYS" -lt 7 ]; then
    echo -e "${YELLOW}WARNING: Latest version ($LATEST) is only $VER_AGE_DAYS days old (threshold: 7)${NC}"
    WARNINGS=$((WARNINGS + 1))
  else
    echo -e "${GREEN}OK: Version $LATEST age: $VER_AGE_DAYS days${NC}"
  fi
fi

# Check downloads > 500
if [ "$DOWNLOADS" -lt 500 ]; then
  echo -e "${YELLOW}WARNING: Only $DOWNLOADS weekly downloads (threshold: 500)${NC}"
  WARNINGS=$((WARNINGS + 1))
else
  echo -e "${GREEN}OK: Weekly downloads: $DOWNLOADS${NC}"
fi

echo "---"

if [ "$WARNINGS" -gt 0 ] && [ "$FORCE" != "--force" ]; then
  echo -e "${YELLOW}$WARNINGS warning(s) found. Install anyway? (y/N)${NC}"
  read -r REPLY
  if [ "$REPLY" != "y" ] && [ "$REPLY" != "Y" ]; then
    echo "Cancelled."
    exit 1
  fi
fi

echo "Installing $PACKAGE..."
npm install "$PACKAGE"
echo -e "${GREEN}Done!${NC}"
```

Сделай исполняемым:
```bash
mkdir -p ~/bin
chmod +x ~/bin/npm-safe-install
```

Добавь `~/bin` в PATH (если ещё нет):
```bash

  grep -q 'export PATH="$HOME/bin:$PATH"' ~/.zshrc || echo 'export PATH="$HOME/bin:$PATH"' >> ~/.zshrc

  grep -q 'export PATH="$HOME/bin:$PATH"' ~/.bashrc || echo 'export PATH="$HOME/bin:$PATH"' >> ~/.bashrc

```

## 1.10 GitGuardian (ggshield)

> **Что:** ggshield (GitGuardian) — сканер секретов (secrets scanner). Проверяет код на случайно добавленные пароли, API-ключи, токены доступа.
> **Простыми словами:** Как рамка-металлоискатель на выходе, только наоборот — проверяет, не уносишь ли ты из дома что-то ценное (пароли, ключи) в публичное место (GitHub).
> **Зачем:** Одна из самых частых ошибок — случайно закоммитить файл с паролем или API-ключом. Код попадает на GitHub, бот находит ключ за секунды, и кто-то использует твой аккаунт. ggshield перехватывает это ДО коммита.
> **Связи:** Git (коммиты) → ggshield (проверяет каждый коммит перед отправкой) → GitHub (код уже без секретов). Работает как фильтр между твоим кодом и публичным репозиторием.
> **Альтернативы:** truffleHog, gitleaks — тоже ищут секреты. GitGuardian выбран за бесплатный план для личных проектов и удобный CLI.
> **Когда пригодится:** Ты подключил Supabase, скопировал ключ в код для теста и забыл убрать. Без ggshield — ключ улетает на GitHub, бот находит его за секунды, и кто-то может использовать твой аккаунт. С ggshield — коммит блокируется, ты убираешь ключ в переменную окружения (.env файл — секретные настройки, которые хранятся отдельно от кода и не попадают в Git), и всё безопасно.

Установи:

**macOS:**
```bash
brew install gitguardian/tap/ggshield
```

**Или через pip:**
```bash
pip install ggshield
```

### Авторизация ggshield

**Предупреди участника ПЕРЕД запуском:**

> Сейчас нужно авторизовать GitGuardian. Откроется браузер — нужно создать бесплатный аккаунт (или войти) и разрешить доступ. Готов?

Дождись подтверждения, затем попроси участника выполнить:
```
ggshield auth login
```

Если участник не хочет создавать ещё один аккаунт — это опционально. Отметь в отчёте что ggshield не авторизован.
